The lifecycle nobody tracks

Anatomy of a decision going stale

Follow a single business rule from the moment it's decided to the moment it silently breaks your product.

Day 0 — The Decision

"Suppliers must be validated against the approved list"

Product lead decides this in a meeting. It's in the Slack thread. Maybe someone adds it to Notion. The developers nod.

Week 2 — The Code

A developer implements validateSupplier()

Looks great. Passes review. Ships to production. The decision is now frozen in code. No link back to why it exists or who decided it.

Month 3 — The Drift

The business rule changes. The code doesn't.

Finance decides approved suppliers now also need credit rating checks. They update the policy doc. Nobody tells the codebase. The Slack thread is buried under 4,000 messages.

Month 5 — The New Dev

A new team member reads the code. Trusts the code.

They build a new feature on top of validateSupplier(). It's working perfectly — against rules that no longer exist. Two features now carry the same wrong assumption.

Month 8 — The Bug Report

"Why are unapproved suppliers getting through?"

Someone finally notices. Now you get to play detective: read the code, search Slack, check three wikis, ask that person who left two months ago. The code was never wrong — it was just outdated.

The hidden cost

Code archeology is expensive

When code is your source of truth, understanding why becomes a research project.

🔍

"Why did we build it this way?"

New developer reads the code. Code doesn't explain the business reason. They spend 2 hours in git blame, Slack search, and Confluence before finding a half-answer in someone's personal notes.

~2-4 hours per archeology session
🔄

"Oops, that's not how we do it anymore"

AI assistant reads your code to learn the patterns. Your code says "vendor." Your business now says "supplier." The AI confidently generates code using the wrong terminology. You re-run. And re-run.

~3 re-runs per stale assumption
🤖

"The AI built exactly what I asked for"

It did. Based on code patterns that were correct 6 months ago. The approval threshold changed, the naming convention changed, the table schema changed. The AI doesn't know. It read your code. Your code is wrong.

~1-2 days of rework per stale feature
👻

"Don't touch that code, nobody knows why it's there"

The scariest line in any codebase. A piece of logic nobody dares to change because the person who wrote it left and the business rule it implements is undocumented. Technical debt isn't always about code quality — sometimes it's about lost context.

Permanent drag on velocity
📈

"Our senior devs spend 30% of their time explaining context"

The tribal knowledge problem. Your best engineers become walking documentation. They can't focus on building because they're constantly fielding "why" questions. When they leave, the knowledge leaves with them.

30% of senior dev time
⚠️

"We shipped a bug because the spec changed and nobody updated the code"

The spec lived in a Google Doc. The code lived in GitHub. The business rule lived in someone's head. When the rule changed, the doc was updated. The code wasn't. Months later: production incident.

Incident response + postmortem + rework

"Code tells you what the system does. It doesn't tell you why, when that decision was made, or whether it's still true. That's the gap where bugs, rework, and 'wait, why?' live."
